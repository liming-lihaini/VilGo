# ADR: 公示状态处理方案

## 背景

Issue #7 村委公示栏要求公示具有状态（未过期/已过期），需要在 expire_date 到期后自动更新状态。需确定状态计算方式。

## 备选方案

### 方案 A: 查询时计算

- 优势: 无需额外字段存储状态；状态实时准确，不存在延迟；无需定时任务资源；实现简单
- 劣势: 每次查询需计算状态；有 expire_date 字段可轻松计算；SQLite 小数据量无性能问题

### 方案 B: 定时任务批量更新

- 优势: 查询性能略优（状态已存储）
- 劣势: 需要定时任务调度，增加系统复杂度；状态更新存在延迟；需要额外的调度框架或线程；增加运维成本

## 决策

选择方案 A（查询时计算），因为 MVP 阶段优先保证实现简单和代码可维护；公示数据量小，查询时计算无性能影响；项目目前无定时任务基础设施，新增会增加复杂度；Boundary 已明确"公示自动过期提醒"范围之外，无需定时任务。

## 否决项

- 方案 B: 项目当前无定时任务需求，引入增加不必要的复杂度；状态计算逻辑简单，查询时计算足够

## 影响

- notices 表不存储 status 字段（或存储但查询时忽略）
- Service 层在返回公示列表时根据 expire_date 判断状态
- 前端根据 expire_date 显示状态标签
