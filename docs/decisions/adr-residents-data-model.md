# ADR: 村民档案数据表设计

## 背景

Issue #1 村民档案管理需要持久化存储村民基础信息，包含身份证号（唯一标识）、姓名、性别、出生日期、联系方式、家庭住址、婚姻状况、政治面貌、保障类型、人员类型、人员角色、销户状态等字段。采用 SQLite 本地数据库，需设计合理的表结构、索引和约束。

## 备选方案

### 方案 A: 单表设计（residents）

- 优势: 简单直接，查询效率高，符合项目 MVP 阶段快速迭代需求；与 MyBatis-Plus 单表操作模式天然契合；无需跨表关联，事务一致性好维护
- 劣势: 未来扩展困难，如需关联家庭成员、社保记录等需另建表并修改现有逻辑；字段过多时影响查询性能（但 SQLite 支持良好）

### 方案 B: 主从表设计（residents + resident_extend）

- 优势: 字段解耦，便于扩展；核心信息与扩展信息分离
- 劣势: 增加查询复杂度，需 JOIN 操作；MVP 阶段过早优化，增加开发维护成本

## 决策

选择方案 A（单表设计），因为 MVP 阶段优先保证开发效率和维护简单性；村民档案字段数量在 20 个以内，SQLite 处理无压力；项目采用前后端一体化架构，单表操作更符合 MyBatis-Plus 的最佳实践。

## 否决项

- 方案 B: MVP 阶段过早优化，增加不必要的复杂度；当前字段设计已覆盖 F-RES-001~007 所有需求

## 影响

- 新建 `src/main/java/com/village/entity/Resident.java` 实体类
- 新建 `src/main/java/com/village/dao/ResidentDao.java` 数据访问层
- 数据库自动创建 `residents` 表（含 id, id_card UNIQUE 约束, deleted 软删除字段）
