# ADR: 公示类型管理方案

## 背景

Issue #7 村委公示栏功能需要管理公示类型（通知/政策文件/财务公开/项目公示/惠民信息）。需要确定公示类型的存储和管理方式。

## 备选方案

### 方案 A: 枚举预置（代码硬编码）

- 优势: 实现简单，无需额外数据库表；查询效率高，无 IO 开销；类型固定时维护成本低
- 劣势: 修改类型需要修改代码并重新部署；不符合灵活配置的业务趋势；与项目已有的数据库驱动配置模式不一致

### 方案 B: 数据库配置表

- 优势: 支持动态配置，无需修改代码；符合项目统一的数据管理规范（其他模块如 special_groups、public_activities 也使用数据库）；便于后期扩展类型属性（如排序、图标）
- 劣势: 增加一次数据库查询；需要维护配置表数据

## 决策

选择方案 B（数据库配置表），因为项目已有 similar patterns（如 special_groups 的 group_type、public_activities 的 activity_type 都使用 VARCHAR + 数据库存储）；MVP 阶段仍需保持灵活性以应对需求变化；查询一次数据库的开销在可接受范围内。

## 否决项

- 方案 A: 硬编码不符合项目数据管理规范；修改类型需要重新部署，与前后端一体化部署模式冲突

## 影响

- 新建 `notice_types` 配置表或在 notices 表中使用 VARCHAR 存储类型
- Service 层需处理类型的下拉列表获取（固定返回 5 种类型）
