# 交付验收阶段知识

## 1. 阶段目标

完成功能开发后，进行**验收测试**、**PR创建**、**知识沉淀**，确保产出可交付。

---

## 2. 交付内容

### 2.1 代码产出

| 类型 | 说明 |
|------|------|
| 后端代码 | Controller / Service / Repository / Model |
| 前端代码 | Views / Components / API / Router |
| 数据库 | schema.sql 变更、迁移脚本 |
| 配置 | application.yml 等配置文件 |

### 2.2 文档产出

| 类型 | 说明 |
|------|------|
| ADR | docs/current/decisions/ 下新增的决策文档 |
| 实现计划 | docs/plans/ 下的计划文档 |
| 测试记录 | 测试用例和结果 |

---

## 3. 验收流程

### 3.1 功能验收

对照 Issue 的**验收条件**逐项验证：

| 步骤 | 操作 |
|------|------|
| 1 | 启动应用 (mvn spring-boot:run) |
| 2 | 按照验收条件逐项测试 |
| 3 | 记录每项的测试结果 (通过/失败) |
| 4 | 修复失败的验收项 |
| 5 | 重新测试直至全部通过 |

### 3.2 架构合规检查

| 检查项 | 标准 |
|--------|------|
| 分层约束 | Controller → Service → Repository → Model |
| 命名规范 | Java: UpperCamelCase, 方法: lowerCamelCase |
| 响应格式 | {code, message, data} 统一格式 |
| 异常处理 | Service 层抛异常，Controller 层统一捕获 |

### 3.3 安全合规检查

| 检查项 | 标准 |
|--------|------|
| SQL注入 | 使用参数化查询，禁止字符串拼接 |
| 路径穿越 | 防止 ../ 攻击 |
| 文件上传 | 禁止上传可执行文件 (.exe/.bat/.jar) |
| 敏感信息 | 禁止硬编码密码/密钥 |
| 操作日志 | 敏感操作需记录日志 |

## 4. PR 创建规范

### 4.1 PR 标题格式

```
{类型}: {简短描述}

示例:
feat(search): 实现全文检索功能
fix(trash): 修复回收站恢复失败问题
```

类型前缀:
- `feat`: 新功能
- `fix`: 修复
- `refactor`: 重构
- `docs`: 文档
- `chore`: 杂项

### 4.2 PR 描述模板

```markdown
## 变更摘要
简要描述本次变更的内容

## 变更模块
- 模块A: 变更内容
- 模块B: 变更内容

## 知识变更
如有知识文件更新，在此说明

## 测试结果
- [ ] 功能测试: 通过/失败
- [ ] 接口测试: 通过/失败
- [ ] 手动测试: 通过/失败

## 关联 Issue
Closes #N

---
Co-Authored-By: Claude Opus 4.7 <noreply@anthropic.com>
```

### 4.3 提交规范

| 规范 | 要求 |
|------|------|
| 提交信息 | 简洁清晰，说明"做了什么" |
| 提交粒度 | 合理分组，避免超大分支 |
| 无关文件 | 禁止提交无关更改 |

## 5. 本阶段执行标准

### 5.1 输入要求

| 输入项 | 来源 | 说明 |
|--------|------|------|
| Issue 描述 | docs/issues/issue-{N}-*.md | 验收条件 |
| 实现产出 | 代码文件、ADR、计划文档 | implement 阶段的产出 |
| 验收条件 | Issue 中的验收条件列表 | 逐项对照验证 |
| pipeline-state | .claude/pipeline-state-{N}.md | 当前状态 |

### 5.2 执行步骤

1. **功能验收** - 对照验收条件逐项测试
2. **架构合规** - 检查分层约束和命名规范
3. **安全合规** - 检查 SQL 注入、路径穿越等
4. **代码自检** - 确保代码质量和可读性
5. **创建 PR** - 按照规范创建 Pull Request
6. **更新 State** - 标记 deliver 完成

### 5.3 输出要求

| 输出项 | 内容 |
|--------|------|
| PR 链接 | GitHub/GitLab PR 地址 |
| 验收结果 | 每项验收条件的测试结果 |
| 遗留问题 | 无法解决的问题清单 |

### 5.4 验收条件

| 条件 | 判定标准 |
|------|----------|
| 功能验收通过 | 所有验收条件测试通过 |
| 架构合规 | 分层、命名、响应格式符合规范 |
| 安全合规 | 无 SQL 注入、路径穿越等安全问题 |
| PR 规范 | 标题、描述、提交信息符合规范 |
| 变更范围 | 仅包含 Issue 相关的文件变更 |

## 6. 测试要求

### 6.1 功能测试

| 测试类型 | 说明 |
|----------|------|
| 单元测试 | 核心 Service 方法测试 |
| 集成测试 | API 接口测试 |
| 手动测试 | 端到端功能验证 |

### 6.2 测试用例模板

```markdown
### 用例N: {功能点}

**前置条件**: {测试前需满足的条件}
**测试步骤**:
1. 执行操作A
2. 执行操作B
3. 验证结果

**预期结果**: {期望的输出}
**实际结果**: {实际输出}
**状态**: 通过/失败
```

### 6.3 边界测试

| 边界场景 | 测试点 |
|----------|--------|
| 文件大小 | 0字节、100MB、超大文件 |
| 文件名 | 空名称、50字符、特殊字符 |
| 目录深度 | 根目录、10层嵌套 |
| 并发操作 | 多任务同时执行 |

## 7. 常见问题

### 7.1 验收失败

| 问题 | 处理方式 |
|------|----------|
| 功能缺陷 | 修复代码后重新测试 |
| 边界问题 | 补充边界测试用例 |
| 性能问题 | 优化后重新测试 |

### 7.2 合规问题

| 问题 | 处理方式 |
|------|----------|
| 分层违规 | 重构代码到正确层级 |
| 安全漏洞 | 修复安全问题 |
| 命名不规范 | 统一命名风格 |
