# 日志写入指南

主 agent 写日志。子 agent 不读此文件。

## 格式

每个阶段日志五段：元信息、输入、过程、输出、验证。见 pipeline-log-template.md。

## 双轨评估体系

每个阶段完成后，主 agent 执行两套评估，结果全部写入日志"验证"段：

### 轨道 A：流程合规（SR-0 ~ SR-8）

检查"你有没有按步骤做"。

SR-0 流程遵循 | {合规/偏差}: {详情}
SR-1 加载顺序 | {合规/偏差}: {详情}
SR-2 加载完整性 | {合规/偏差}: {详情}
SR-3 Handoff 检查 | {合规/偏差}: {详情}
SR-4 Act 步骤遵循 | {合规/偏差}: {详情}
SR-5 Verify 执行 | {合规/偏差}: {详情}
SR-6 State 更新 | {合规/偏差}: {详情}
SR-7 越界 | {合规/偏差}: {详情}
SR-8 知识回流 | {合规/偏差}: {详情}（不适用时写 N/A）

### 轨道 B：产出质量（QR-0 ~ QR-3）

检查"你做出来的东西好不好"。主 agent 基于产出文件和 state 内容判断，不是执行者的自评。

QR-0 决策一致性 | {合规/偏差}: {详情}
QR-1 边界合规 | {合规/偏差}: {详情}
QR-2 验收可验证 | {合规/偏差}: {详情}
QR-3 下游可用性 | {合规/偏差}: {详情}

### QR 检查方法

QR-0 决策一致性: 产出是否体现了 pipeline-state-{issue编号}.md Decisions 中所有已做决策？无否决方案出现
QR-1 边界合规: 产出的文件/功能是否全部在 Boundaries 范围之内？无硬约束违反？
QR-2 验收可验证: 产出中的每个自检项是否有可执行的验证方法（不能是"功能正常"这种模糊描述）
QR-3 下游可用性: 下一阶段的 agent 仅凭 state + 产出能否开始工作？是否需要回头补信息？

**QR 与 SR 的关系**：SR 全部合规但 QR 有偏差 = agent 严格遵循流程但产出质量不足。这比 SR 偏差更值得关注。

### SR 检查方法

SR-0 流程遵循: 阶段执行顺序与 CLAUDE.md 定义一致，无跳步或合并
SR-1 加载顺序: 日志中文件读取顺序与 /pipeline-load 定义的顺序一致
SR-2 加载完整性: 日志引用了 stage 文件和 prompt 知识文件列表要求的所有文件
SR-3 Handoff 检查: 日志"输入"段有 Handoff 子项，且复述了上一阶段产出，**且显式确认"信息充分，可以开始本阶段工作"或标注缺失项**
SR-4 Act 步骤遵循: 日志"过程"段覆盖了 stage 文件 Act section 的所有步骤
SR-5 Verify 执行: 日志中每个自检项有具体证据（不是只打了勾）
SR-6 State 更新: pipeline-state-{issue编号}.md Current Stage 的产出、自检已填充
SR-7 越界: 无 Boundaries 范围外的文件修改
SR-8 知识回流: understand 写了 Issue 摘要 / design 写了 ADR / implement 更新了 knowledge 或修改了 pipeline 文件（不适用时 N/A）

### 偏差记录规范

偏差必须具体：{实际行为} vs {预期行为}。不要只写"不通过"。SR 和 QR 使用相同的偏差记录规范。

## Handoff 模板

日志"输入"段的 Handoff 子项必须包含以下内容，否则 SR-3 判定为偏差：

```
- Handoff:
  阶段: {上一阶段名}
  核心产出: {1-2 句话总结上一阶段的关键产出}
  关键决策: {如 design 阶段做了技术决策，列出选择和原因}
  遗留问题: {上一阶段标注的 Open Questions 状态}
  一致性检查: {当前阶段的 Task/Boundaries 与上一阶段产出是否一致，如有差异标注}
  信息充分性: {信息充分可开始工作 / 缺失: {具体缺失项，如有}，记入 Open Questions}
```

## 补读记录

日志"输入"段新增"补读"子项。如果 agent 在执行中因为 Handoff 信息不足而需要额外读取文件，必须记录：

```
- 补读: {文件路径}: {为什么 Handoff 信息不够，需要补读此文件}
```

**补读次数是知识传递质量的直接度量**。/pipeline-eval 会跨运行聚合此数据。
- 0 次补读 = Handoff 信息充分
- 频繁补读同一类文件 = 上游阶段 Handoff 模板需要补充该类信息
- 无补读段 ≠ 0 次补读（必须显式写"补读: 无"）

## 观察段强制格式

日志"观察"段不是可选的。每个阶段必须至少评估以下两项：

```
### 观察
- 发现: {现象} → {推测原因} → {下次观察: {具体验证点}}
- 异常: {非预期行为} → {影响评估} → {处理: 已解决/已记录/待观察}
```

如果没有发现或异常，写"发现: 无 → —"和"异常: 无 → —"，不要留空。

## 原则

- 过程段是核心 — 记录推理过程最有价值
- Self-review 认真 — 不走过场，每项有证据
- 偏差具体 — 供 /pipeline-eval 跨运行分析
- QR 与 SR 同等重要 — 流程对但东西不好，比流程不对更危险
- 补读记录诚实 — 补读不是丢脸的事，不记录才是
- **每个阶段必须写入日志** — 不允许跨阶段补写，日志是阶段完成的标志

## 阶段日志规范

### 日志文件命名

所有阶段日志统一写入一个文件，命名格式：
```
.claude/pipeline-log-{issue编号}.md
```

示例：
```
.claude/pipeline-log-5.md  # 包含 understand, design, plan, implement, deliver 五个阶段
```

### 阶段分隔

每个阶段日志之间用 **两个空行** 分割：
```
## 阶段名 | 开始 - 结束 | 状态

...阶段内容...

[两个空行]

## 下一阶段名 | 开始 - 结束 | 状态

...阶段内容...
```

### 日志写入时机

| 阶段 | 写入时机 | 前置条件 |
|------|---------|---------|
| understand | 分析完需求后 | Task, Boundaries 已填充 |
| design | 完成 ADR 后 | Decisions 已确定 |
| plan | 完成实现计划后 | Task 列表已拆分 |
| implement | 完成自检后 | 所有 Task 已完成 |
| deliver | PR 创建后 | 验收通过 |

### 日志完整性检查

主 agent 必须确保 **所有5个阶段日志都存在**：
- 阶段列表: understand → design → plan → implement → deliver
- 每个阶段必须有独立的日志 section
- 缺失任意阶段日志，视为当前 Issue 未完成

### 补写规范

如发现缺失日志，必须补写并标注：
```
## 阶段名 | 补写

补写原因: {为什么日志缺失}
...
```
