# 自我进化规范

Pipeline 完成后执行。仅迭代 pipeline 文件自身，不涉及代码质量反馈。

## 指标体系

### 核心指标（6 个）

| # | 指标 | 计算 | 退化动作 |
|---|------|------|---------|
| 1 | 一次通过率 | 不需修复的阶段 / 总阶段数 | 低 → 检查失败阶段的 stage 文件指令是否清晰 **或评审标准是否太松** |
| 2 | 评审发现率 | review 发现阻塞问题数 / review 次数 | 零 → 区分"评审太松"还是"产出质量高"。**校准方法**：人工抽查 1-2 个通过的 review，验证是否真的无需改进 |
| 3 | SR 合规率 | SR-0~SR-8 合规项 / 总检查项 | 低 → 运行 /pipeline-eval 定位系统性偏差 |
| 4 | QR 合规率 | QR-0~QR-3 合规项 / 总检查项 | 低 → 分析哪个 QR 项最常偏差。**注意**：QR 偏差比 SR 偏差更严重（流程对但产出差） |
| 5 | 补读率 | 有补读的阶段数 / 总阶段数（非首阶段） | 高 → 分析补读原因分类，修改上游 Handoff 模板 |
| 6 | 总耗时 | understand → deliver | 持续增长 → 分析 per-stage 耗时表，定位增长最快的阶段 |

### 交叉分析（每次必须做）

| 分析 | 方法 | 目的 |
|------|------|------|
| 一次通过率 vs 评审发现率 | 两个指标同时低 → 评审标准合理但产出质量差；通过率高+发现率低 → 可能评审太松 | 校准评审严格度 |
| SR 合规率 vs QR 合规率 | SR 高+QR 低 = 流程合规但产出质量不足（最危险的组合） | 识别"走过场" |
| 补读率 vs 一次通过率 | 补读率高+一次通过率也高 → 补读机制在弥补 Handoff 不足，但掩盖了上游问题 | 区分真实质量 vs 补救措施 |
| Per-stage 耗时 vs 质量指标 | 某阶段耗时增长但质量指标未提升 → 该阶段的增量投入无效 | 识别投入产出失衡 |

### 指标校准规则

指标在没有校准的情况下会漂移。以下规则防止漂移：

| 信号 | 校准动作 |
|------|---------|
| 一次通过率连续 2 次 ≥ 95% 且评审发现率 = 0 | 人工抽查评审，确认评审标准是否需要收紧 |
| SR 合规率连续 2 次 = 100% 但 QR 合规率 < 80% | 检查 SR 检查方法是否过于宽松，或 QR 项定义是否合理 |
| 补读率连续 3 次 = 0% | 检查补读记录是否被遗漏（agent 忘了记录 ≠ 没有补读） |

## 退化分析

1. 定位：读退化指标对应阶段的日志"过程"段
2. 分类：
   - 子 agent 不遵循指令 → 更新 stages/{name}.md
   - 知识加载问题 → 更新 commands/pipeline-load.md
   - 交接信息不足 → 更新 pipeline-state-template.md 字段
   - 评审标准不对 → 更新 stages/review.md
   - 编排流程问题 → 更新 CLAUDE.md
   - 产出质量不足（SR 合规但 QR 偏差） → 检查 stage 文件的 Act/Verify 指令是否只管过程不管质量
3. 改进：修改对应文件
4. 验证：标注"下次观察: {具体观察点}"

## 进化路径

Pipeline 版本用 symlink 管理：

1. 修改文件后，创建新版本目录 stages/v{N+1}/
2. 复制 v{N} 全部文件到 v{N+1}，应用修改
3. 更新 stages/current -> v{N+1}
4. 在 Iteration 记录中标注版本号
