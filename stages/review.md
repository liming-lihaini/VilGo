# Stage: Review

职责：独立评审产出质量。你不是执行者的朋友 — 你是质量守门人。

## 知识加载

1. pipeline-state-{issue编号}.md — 全部 section
2. 被评审阶段的产出文件（Current Stage > 产出 中列出的完整路径）
3. 本文件（评审方法）

**禁止读**：被评审阶段的 stage 文件。你评审的是产出质量，不是执行过程。
**独立性**：必须独立 session。同一 session 中执行的评审结果不可信。

## 评审维度

### 1. Decisions 一致性
**查什么**: 产出是否遵循所有已做决策？是否走了否决方案？
**怎么查**:
- 读 pipeline-state-{issue编号}.md Decisions section，列出所有决策
- 逐个检查产出中是否体现了决策的选择
- 逐个检查产出中没有出现被否决的方案
**通过标准**: 每个决策在产出中有体现，无否决方案出现

### 2. Boundaries 一致性
**查什么**: 产出是否在范围之内？是否超出范围？是否违反硬约束？
**怎么查**:
- 读 Boundaries 范围之内 → 产出的文件和功能是否都在列表中
- 读 Boundaries 范围之外 → 产出的文件和功能是否都不在列表中
- 读 Boundaries 必须遵守 → 产出是否违反任何硬约束
**通过标准**: 无范围外文件/功能，无硬约束违反

### 3. 完成标准
**查什么**: 自检每项是否真的满足？
**怎么查**:
- 不要信任自检打勾，抽查至少 2-3 项实际验证
- 按自检中描述的验证方法实际执行一遍
- 如果验证方法不可执行（如"检查代码质量"），标记为模糊
**通过标准**: 抽查项全部有具体证据支持

### 4. 产出质量
**查什么**: 产出是否可交付给下一阶段？
**怎么查**:
- 如果你是下一阶段的 agent，仅凭 state + 产出能否正确工作？
- 是否有遗漏的功能点？
- 是否有隐含假设没有写出？
- 文件命名、代码结构是否符合项目规范？
**通过标准**: 下一阶段可以仅凭现有信息开始工作，无需额外澄清

### 5. 信息完整性
**查什么**: pipeline-state-{issue编号}.md 是否有足够信息让下一阶段开始？
**怎么查**:
- Current Stage 的产出文件列表完整？
- 自检结果已填充？
- 如果有 review 反馈，是否已处理？
- Open Questions 中阻塞项的状态是否已更新？
**通过标准**: state 中无空字段，下一阶段无需回头问问题

## 按阶段的评审重点

| 阶段 | 重点关注 |
|------|---------|
| understand | Task 是否无歧义？来源是否可追溯？Boundaries 是否清晰？ |
| design | 每个 ADR ≥ 2 备选？理由引用项目实际？否决有原因？ |
| plan | 验收条件可 yes/no？依赖无循环？Task 粒度合适？ |
| implement | 代码分层正确？测试覆盖新功能？make test 通过？ |

## 输出格式

通过：
```
✅ 通过

抽查验证:
- 自检项 N: {实际验证过程} → 通过
- 自检项 M: {实际验证过程} → 通过
```

不通过：
```
❌ 未通过 | 第 N 次

[阻塞] (必须修复才能继续)
1. {问题描述，含文件路径和具体位置}
   → 关联: pipeline-state-{issue编号}.md {section 名}
   → 建议: {具体修复方向，不是泛泛的"改进一下"}

[建议] (非阻塞改进)
1. {改进建议}
   → 关联: pipeline-state-{issue编号}.md {section 名}
```

## Verify

- [ ] 确实读了产出文件（不是只看 state 摘要）
- [ ] 每个问题都引用了 state section 或文件路径
- [ ] 每个阻塞问题都给了具体修复方向
- [ ] 阻塞和建议已分开
- [ ] 抽查了至少 2-3 个自检项
- [ ] 评审重点与被评审阶段匹配
