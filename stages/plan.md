# Stage: Plan

职责：把设计决策转化为可执行计划。每个 Task 必须有可检查的验收条件。

## Task 粒度标准

| 粒度 | 标准 | 适合 |
|------|------|------|
| 太粗 | 一个 Task 涉及 3+ 个文件或无法独立验证 | 拆分 |
| 合适 | 一个 Task 可独立验证（参考：通常 1-3 个文件） | ✅ |
| 太细 | 一个 Task 只是改一个字段名或加一个 import | 合并 |

核心标准是**可独立验证**，行数仅作参考。

## 依赖类型

| 类型 | 含义 | 表示 |
|------|------|------|
| 硬依赖 | Task B 必须等 Task A 完成 | 依赖: Task A |
| 软依赖 | Task B 最好在 Task A 之后，但可以并行 | 依赖: Task A (soft) |
| 无依赖 | 可与其他 Task 并行 | 依赖: 无 |

有软依赖的 Task 可以并行执行，但执行顺序建议遵循依赖方向以减少冲突。

## 并行化指导

- **默认串行。** 如果不确定能否并行，标为串行。串行更安全
- 满足以下**全部**条件时可标注可并行：(1) 涉及完全不同的文件 (2) 无数据依赖 (3) 不影响同一文件的不同 section
- 同一文件的不同 section 修改禁止并行（会产生 merge conflict）

## Act

> 执行 `/pipeline-load plan` 进行知识加载

1. 拆分为独立 Task：
   - 每个 Task 有明确边界（不与其他 Task 重叠）
   - 每个 Task 有可检查的验收条件（见下方标准）
   - 每个 Task 标注涉及文件和预估变更量

2. 确定依赖顺序：画出依赖图，确认无循环

3. 标注可并行 Task：如果多个 Task 无硬依赖，标注"可并行"

4. 写执行计划（docs/plans/{YYYY-MM-DD}-{title}.md）在docs目录下创建issue目录，按issue进行归档，格式：
   # 执行计划: {Issue 标题}
   ## Task 1: {标题}
   做什么: {具体描述}
   涉及文件: {预计修改/创建的文件路径}
   预估变更: {约 XX 行}
   验收条件:
     - {可客观判定，yes/no 可回答}
   依赖: 无 / Task N / Task N (soft)

   ## 可并行的 Task
   - Task 2 和 Task 3 可并行（无硬依赖，涉及不同文件）

   ## 不做的事情
   - {排除的工作}: {为什么排除}

5. 更新pipeline-state-{issue编号}.md:
   - Decisions 追加执行顺序决策
   - References 添加执行计划完整路径

## Verify

自检（每项必须有具体证据）：
- [ ] Task 粒度合适：每个 Task 可独立验证，涉及文件数合理
- [ ] 验收条件可检查：逐条可 yes/no 回答（不能是"代码好"、"功能正常"）
- [ ] 无 design 遗漏：对照 Decisions 逐项检查，每个决策有对应 Task
- [ ] 依赖无循环：画出依赖图验证
- [ ] 无 Boundaries 违反：对照范围之外逐项检查
- [ ] 并行标注合理：标注可并行的 Task 确实无硬依赖且涉及不同文件

## 验收条件标准

- ✅ "POST /api/login 返回 200 且 response body 含 token 字段"
- ✅ "单元测试覆盖 login 函数的 3 个分支（手机号/邮箱/参数错误）"
- ✅ "make lint 零错误"
- ✅ "GET /api/tasks/{id} 的 response 包含 checkpoint_count 字段且值为整数"
- ❌ "登录功能正常"
- ❌ "代码质量好"
- ❌ "完成设计"
- ❌ "测试通过"（哪个测试？通过的标准是什么？）

## 常见错误

- ❌ 10 个 Task 全部串行，其中 5 个无依赖 → ✅ 标注可并行
- ❌ 一个 Task 涉及 8 个文件 → ✅ 拆分为多个 Task
- ❌ 验收条件是"功能正常" → ✅ 写出具体的接口行为和预期值
