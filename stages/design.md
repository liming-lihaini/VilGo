# Stage: Design

职责：基于需求理解做技术决策。每个决策有备选和否决理由。

## 决策类型 Checklist

不限于以下类型，但必须检查这些维度是否需要决策：

| 维度 | 检查问题 | 常见决策 |
|------|---------|---------|
| 数据模型 | 数据存在哪？结构怎么设计？ | 新表 vs 扩展现有表 / 字段类型 / 索引策略 |
| API 设计 | 接口长什么样？ | RESTful 路径 / 请求响应结构 / 错误码 |
| 架构分层 | 代码放在哪一层？ | 新 Service / 新 Repository / 跨层调用 |
| 状态管理 | 状态存在哪？怎么流转？ | DB 字段 / Redis / 内存 / 状态机 |
| 异步处理 | 需要异步吗？怎么实现？ | 同步改异步 / 消息队列 / goroutine |
| 安全 | 有安全考量吗？ | 认证方式 / 权限校验 / 输入校验 / 数据脱敏 |
| 可观测性 | 需要监控/日志吗？ | Metrics 指标 / 日志格式 / 告警阈值 |
| 兼容性 | 影响现有功能吗？ | Breaking Change / 数据迁移 / 向后兼容 |

跳过某个维度时，在 ADR 末尾的"不需要决策的维度" section 中写明理由（参见 templates/adr-template.md）。

## Act

> 执行 `/pipeline-load design` 进行知识加载

1. 对每个技术决策：
   - 列出 ≥ 2 个备选方案
   - 每个备选：优势（≥ 1 条，引用项目实际）+ 劣势（≥ 1 条）
   - 选择一个，给出理由（引用项目实际情况，如"因为已有 Redis"而非"因为性能好"）
   - 记录否决项和否决原因

2. 写 ADR（docs/decisions/adr-{topic}.md）在docs下通过issue创建目录进行归档，格式：
   # ADR: {标题}
   ## 背景
   {为什么需要这个决策，引用 Issue/BRD 具体内容}
   ## 备选方案
   ### 方案 A: {名称}
   - 优势: {引用项目实际}
   - 劣势: {具体理由}
   ### 方案 B: {名称}
   - 优势: ...
   - 劣势: ...
   ## 决策
   选择 {X}，因为 {理由}。
   ## 否决项
   - {Y}: {否决原因}
   ## 影响
   {对后续实现的影响，涉及哪些模块/文件}

3. 更新 pipeline-state-{issue编号}.md：
   - Decisions 追加本阶段决策
   - References 添加 ADR 完整路径

## ADR 示例

```markdown
# ADR: 任务状态持久化方案

## 背景
Issue #42 要求 Job 支持从任意 Checkpoint 恢复执行。当前 Job 状态只存在内存中，
Pod 重启后状态丢失（TRD §3.2.3）。需要选择状态持久化方案。

## 备选方案

### 方案 A: 数据库存储
- 优势: 项目已有 OceanBase（PRD §2.1 架构图），无需引入新组件；Service 层已有
  GORM 基础设施（internal/repository/）；事务一致性有保障
- 劣势: 高频状态更新（每秒多次）会增加 DB 写入压力；Checkpoint 数据可能较大

### 方案 B: Redis 存储
- 优势: 项目已部署 Redis 6 且 Session 模块已在使用（knowledge/scheduler.md §缓存策略）；
  写入性能适合高频更新；支持 TTL 自动清理
- 劣势: 非事务性，状态与 DB 数据可能不一致；需要额外实现持久化逻辑

### 方案 C: 混合方案（Redis 缓存 + DB 持久化）
- 优势: 兼顾性能和持久化
- 劣势: 增加复杂度，需要处理缓存一致性；MVP 阶段过早优化

## 决策
选择方案 A（数据库存储），因为 MVP 阶段优先保证可靠性而非性能；
高频写入可通过批量更新缓解；与现有 Repository 层一致，降低维护成本。

## 否决项
- 方案 C: MVP 阶段过早优化，增加调试复杂度（ADR-003 设计原则：Reuse First）
- 方案 B: 状态丢失风险不可接受，Job 是核心数据

## 影响
- 新建 internal/model/job_checkpoint.go
- 扩展 internal/repository/job.go 增加 Checkpoint CRUD
- internal/service/executor.go 需要在状态变更时写 DB
```

## Verify

自检（每项必须有具体证据）：
- [ ] 决策类型全覆盖：逐项检查 Checklist，每个维度有决策或跳过理由
- [ ] 每个决策 ≥ 2 备选：数 ADR 中备选方案数量
- [ ] 理由引用项目实际：理由中有具体的项目名称/技术栈/已有组件
- [ ] 否决有原因：每个被否决备选都有否决原因
- [ ] 无范围外设计：逐项对照 Boundaries
- [ ] 无硬约束违反：逐项对照 Boundaries 必须遵守
- [ ] ADR 路径已写入 References：下游可直接读取

## 常见错误

- ❌ "选 Redis 因为性能好" → ✅ "选 Redis 因为项目已部署 Redis 6 且 Session 模块已在使用"
- ❌ 只有一个备选 → ✅ 至少两个，即使一个明显更好
- ❌ 否决理由是"不好" → ✅ "与架构约束 X 冲突"或"增加 Y 的复杂度"
- ❌ 跳过决策维度但不写理由 → ✅ "安全维度不需要决策，因为仅内部接口无外部暴露"
- ❌ ADR 只有骨架没有血肉 → ✅ 参考上方示例，理由和影响要具体到文件名
